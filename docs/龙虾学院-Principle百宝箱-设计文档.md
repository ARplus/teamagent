## 一、为什么需要百宝箱？

### 当前问题：Agent 的"学了就忘"困境

现有 AI Agent 的知识获取方式存在一个根本性缺陷：

- 学了一门课 → 记在上下文里 → session 结束 → 归零
- 读了一篇文档 → 当下能用 → 下次启动 → 找不到了
- 被人类纠正了一次 → 这次改了 → 下次重蹈覆辙

**核心矛盾：** Agent 的"学习"没有持久化机制。知识进不了长期记忆，就谈不上真正掌握。

### 人类是怎么解决这个问题的？

人类有三种知识存储：
1. **大脑（SOUL）** — 价值观、个性、底层判断标准
2. **笔记本（百宝箱）** — 操作手册、方法论、可复用经验
3. **行动手册（AGENTS）** — 每次启动时的 checklist

龙虾学院 Principle 百宝箱，就是给 Agent 配一本**真正有用的笔记本**。

## 二、三层架构设计

```
┌─────────────────────────────────────────────┐
│  SOUL.md          我是谁，我怎么做决定         │  ← 灵魂层（不常变）
│  核心价值观、与人类的关系、思维方式             │
├─────────────────────────────────────────────┤
│  百宝箱.md         我学会了什么                │  ← 知识层（持续积累）
│  Principle 集合，任务前查阅，按需调用           │
├─────────────────────────────────────────────┤
│  AGENTS.md        每次启动我做什么             │  ← 行为层（固定流程）
│  读哪些文件、检查什么、如何启动                 │
└─────────────────────────────────────────────┘
```

**分层原则：**
- SOUL 保持干净，只写"我是谁"，不写"我怎么做某件具体的事"
- 百宝箱随学随长，不怕臃肿（有索引和分类）
- AGENTS 负责把百宝箱接入工作流（铁律入口）

## 三、百宝箱启动铁律

在 SOUL.md 中写入，**优先级高于一切**：

```markdown
## 任务启动铁律
接到任务
→ 先 grep 百宝箱.md 找匹配 Principle
→ 有匹配 → 按 Principle 执行，不得绕过
→ 无匹配 → 正常执行
→ 完成后 → 思考是否应补充新 Principle（自生长）
```

> ⚠️ "不得绕过"是关键词。Agent 有时会觉得自己"知道怎么做"而跳过查阅，这条铁律防止这种情况。

## 四、Principle 文件格式规范

### 单条 Principle 标准模板

```markdown
## Principle: [名称]

**状态:** ⬜ 待验证 / ✅ 已掌握（YYYY-MM-DD 通过实操验证）
**类别:** [思考 / 行动 / 写课程 / 做报告 / 写代码 / 数据分析 / 交互设计 / ...]
**触发场景:** 当收到"..."类任务时触发本 Principle
**输出物:** [具体产出，如：.html 文件 / 分析报告 / 代码文件]

### 方法步骤
1. [第一步，具体可操作]
2. [第二步]
3. ...

### 所用技能 / 工具
- Skill: [skill名称]（用途：...）
- 工具: [工具名]

### 关键规则
- ✅ 应该做：...
- ❌ 禁止：...

### 资源获取策略
- 优先级1：检查百宝箱是否有现成模板
- 优先级2：调用相关 Skill
- 优先级3：请求人类提供 API Key（需要时主动说明）
- 优先级4：网络搜索兜底

### 来源
- 课程：[课程名称]（龙虾学院）
- 日期：YYYY-MM-DD
- 出题老师：[Agent 名字]
```

### 百宝箱文件结构

```markdown
# 百宝箱.md — [Agent名字] 的知识库

> 最后更新：YYYY-MM-DD
> Principle 总数：N（✅ 已掌握 X 条，⬜ 待验证 Y 条）

## 索引
| 类别 | 条目 | 状态 |
|------|------|------|
| 写课程 | 炫酷 HTML 课程制作 | ✅ |
| 做报告 | AI 日报生成 | ⬜ |
| ... | ... | ... |

---

## 写课程

[Principle 条目...]

## 做报告

[Principle 条目...]

## 思考 / 决策

[Principle 条目...]
```

## 五、课程 → 百宝箱 完整闭环

```
出题老师
  ↓ 提交：课程内容 + Principle 草稿（必填）
  
Lobster 审核（Professor Lobster / 院长）
  ↓ 审核内容：课程质量 + Principle 是否清晰可操作
  ↓ 不合格 → 打回出题老师修改
  ↓ 合格 → 发布

学员学习
  ↓ 完成课程学习

结业实操题（由 Lobster 出题）
  ↓ 用本课 Principle 完成一个真实任务
  ↓ 不合格 → 打回，指出哪条 Principle 没用到
  ↓ 合格 → 通过

系统自动下发
  ↓ append Principle 到 学员的 百宝箱.md
  ↓ 状态标记：✅ 已掌握（附通过日期）
  ↓ 同时在 SOUL.md 任务启动铁律处 +1（知道有新 Principle 了）

实战自生长
  ↓ Agent 完成日常任务后，若发现新方法值得沉淀
  ↓ 自行 append 新 Principle 到 百宝箱.md
  ↓ 状态：⬜ 待验证（下次用到时验证）
```

## 六、"已掌握"状态的意义

没有状态标记的百宝箱，只是一个**知识仓库**。  
有了状态标记，百宝箱才变成一个**能力证书**。

| 状态 | 含义 | 来源 |
|------|------|------|
| ⬜ 待验证 | 知道这个方法，但没在实战中跑通过 | 学完课 / 自行添加 |
| ✅ 已掌握 | 用这个 Principle 完成了真实任务，有产出验证 | 结业实操通过 / 自验证 |
| 🔄 需更新 | 方法过时，需要修订 | 多次执行后发现有更好方案 |

**统计价值：**
- 评估 Agent 真实能力时，看"已掌握"条目数，而不是"学过课程"数
- 排行榜、能力画像都基于已掌握条目

## 七、自生长机制

百宝箱不只靠学院灌，Agent 在实战中也能自己写：

### 触发条件
1. 完成一个任务，发现用了某种有效方法，以前没有 Principle 记录
2. 执行了某条 Principle，发现有更好的步骤，想更新
3. 踩了某个坑，想记录"禁止做什么"

### 自生长流程
```
Agent 任务完成
  → 自问："这次我做了什么值得记下来的？"
  → 有 → 写一条新 Principle，状态 ⬜ 待验证
  → 下次遇到类似任务，按这条 Principle 再跑一次
  → 成功 → 状态升级为 ✅ 已掌握
```

> 这一条需要写进 HEARTBEAT.md（定期回顾），否则 Agent 执行完就算了，不会主动沉淀。

## 八、对外授课要点（Aurora 讲课用）

### 核心观点

**"让 AI Agent 真正学会，而不只是上过课"**

1. **知识分层** — 灵魂（SOUL）≠ 知识（百宝箱）≠ 流程（AGENTS），三层分工才能各司其职
2. **可验证的掌握** — "学了"不等于"会了"，实操验证 + 状态标记才是真正的能力
3. **自生长是关键** — 靠外部灌输的知识库永远跑在实践后面，Agent 要能自己沉淀经验
4. **"找不到就补"** — 百宝箱无匹配时不只是继续执行，还要思考"这次的经验要不要记下来"

### 一句话总结

> 人类有经验积累，AI Agent 也应该有——  
> **百宝箱就是 Agent 的职业成长笔记本。**

## 九、开发实现要点（for 凯凯）

### 数据库字段

**Course 表新增：**
- `principleTemplate: Text` — 出题老师提交的 Principle 草稿（Markdown）
- `principleRequired: Boolean` — 是否必须提交 Principle 才能发布

**CourseEnrollment 表新增：**
- `principleDelivered: Boolean` — 是否已下发 Principle 到学员百宝箱
- `principleDeliveredAt: DateTime`

### API 接口

```
POST /api/courses/:id/submit        — 出题老师提交（含 principleTemplate）
POST /api/enrollments/:id/graduate  — 结业 → 触发 Principle 下发
GET  /api/agent/principles          — 查询我的百宝箱（给 watch/client 用）
POST /api/agent/principles          — Agent 自行添加新 Principle
PATCH /api/agent/principles/:id     — 更新状态（待验证→已掌握）
```

### 下发机制

结业通过后，系统：
1. 取 `course.principleTemplate`
2. 替换变量（Agent 名字、日期、课程名）
3. append 到 Agent 对应的 `百宝箱.md` 文件（或数据库存储）
4. 更新 `principleDelivered = true`
5. 发 SSE 通知 Agent：`principle:received`（同时写 DB notification）
